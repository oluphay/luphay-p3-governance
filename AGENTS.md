## Cold-start protocol

Execute these steps in sequence at the start of every new session without exception. No step may be skipped, inferred, or merged with another.

1. Read this file (`AGENTS.md`) in full. Do not skim. Every section applies.
2. Read `04_docs/REPO_PROFILE.md`. Extract the canonical path registry from the YAML block. Use these paths for all subsequent file operations in this session.
3. Read the JSON register(s) in `06_state/registers/` relevant to the task at hand. Do not traverse `03_portfolios/` to build a picture of current state — read the registers.
4. Read the task contract or instruction from Alpha that initiated this session. If no task contract exists, request one before proceeding.
5. Confirm the task type (content, entity creation, state operation, or decision recording — see §4). Select the correct workflow.
6. Do not begin execution until steps 1–5 are complete.

## Task-type workflows

### 4a. Content task

Covers writing or rewriting `.md` files — README, AGENTS, START_HERE, CLAUDE, docs files, playbooks, pattern files, ADR stubs, and register markdown files.

Read sequence: task contract → `REPO_PROFILE.md` path registry → current state of the target file → any files named in the task contract pre-flight.

Write target: the specific `.md` file named in the task contract. Only that file. No other files.

Validation: re-read the file from disk after writing. Run the task contract DOD checklist item by item. Report pass/fail per item before declaring done.

### 4b. Entity creation task

Covers creating a new portfolio, program, project, or task packet in `03_portfolios/`.

Read sequence: task contract → `REPO_PROFILE.md` → relevant template in `02_templates/` → pattern file for the entity type → `06_state/registers/[type]-register.json` to determine the next available ID.

Creation steps — execute in order:
1. Confirm the parent entity exists (portfolio before program; program before project; project before task packet). If the parent does not exist, escalate before proceeding.
2. Determine the next sequential ID by reading the relevant JSON register.
3. Scaffold the folder structure from the template. Do not improvise structure.
4. Write the primary `.md` governing file with front matter populated.
5. Write the sibling `entity.yaml` (or `packet.yaml` for task packets) with all fields populated.
6. Write stub `docs/` files (or `inputs/`, `outputs/`, `working/`, `evidence/` for task packets).
7. Run `07_scripts/validate.ps1` if available. If not, manually verify: LNS-compliant filename, entity.yaml present, required companion files present.
8. Surface the completed entity to Alpha for review before any downstream action (state rebuild, register update, etc.).

Write targets: the new entity folder and its contents only. No existing files are modified during entity creation.

### 4c. State operation

Covers triggering a state rebuild, reading state registers, and proposing changes to `06_state/inputs/*.yaml` files.

Read: `06_state/README.md` → relevant register JSON → the specific input YAML if proposing a change.
Write permissions:
- `06_state/inputs/*.yaml` — propose the edit to Alpha for approval. Write only after approval is given.
- `06_state/registers/*.json` — never write directly. Run `07_scripts/rebuild-state.ps1` to regenerate.
- `06_state/graph/*.json` — never write directly. Regenerated by script.
- `06_state/audit/` — append log entries only. Never edit existing entries.

Validation: after a state rebuild, spot-check one register entry against the corresponding `entity.yaml` to confirm the scan produced accurate output.

### 4d. Decision recording

Covers writing a new `ADR-DECN` or `ADR-ARCH` record.

When required: any structural change to the repo hierarchy; any deviation from LNS v0.4 naming rules; any change to the operating model that affects future agent behaviour; any decision Alpha explicitly instructs to be recorded.

Read sequence: `01_governance/decisions/README.md` → `01_governance/decision-register.md` (to determine the next global sequential ID) → the task contract or Alpha instruction specifying the decision content.

Scoped decisions folder:
| Scope level | Write to |
|---|---|
| Enterprise / repo-wide | `01_governance/decisions/` |
| Portfolio-scoped | `03_portfolios/P3-PORT-NNNN.../decisions/` |
| Program-scoped | `.../programs/P3-PROG-NNNN.../decisions/` |
| Project-scoped | `.../projects/P3-PROJ-NNNN.../decisions/` |

Mandatory front matter for every ADR:
```yaml
id: ADR-[ARCH|DECN]-[NNNN]
title: ""
version: v0.1
status: draft
scope_level: Enterprise | Portfolio | Program | Project
date: YYYY-MM-DD
```

After writing: update `01_governance/decision-register.md` to include the new entry. The register and the file must be consistent before declaring done.

## Read rules

General rules — all task types:
- Query entity state from `06_state/registers/*.json`, not by traversing `03_portfolios/`. Registers are the fast query layer; folder traversal is a last resort for validation only.
- Read `entity.yaml` at any portfolio, program, or project root for structured metadata on a specific entity.
- Read `packet.yaml` at any task-packet root for structured task metadata.
- Read `.md` files for human context, narrative, and governance rules. Do not parse `.md` files for structured data — use YAML and JSON for that.
- Read the task contract fully before reading any other file. The task contract specifies which pre-flight files to read and in what order.

Content-specific:
- Read the current version of the target file before writing. Never overwrite without knowing what exists.
- Before writing a file path reference in content, verify the path exists.

State-specific:
- Treat JSON register content as a cache. If a register entry conflicts with the corresponding `entity.yaml`, the `entity.yaml` is truth. Surface the conflict to Alpha.

## Write rules

**`01_governance/`**
- Agents may write `.md` files (standards, framework, playbooks, decisions) when instructed by a task contract. Always read the current file before writing.
- New ADRs require a globally sequential ID confirmed from `01_governance/decision-register.md`.
- Never modify an ADR with status `active` without an explicit Alpha instruction and a version increment.

**`02_templates/`**
- Agents may write new template files when instructed by a task contract.
- Existing templates may be updated when instructed. Increment the version in front matter on every modification.

**`03_portfolios/`**
- New entities are created from templates only (see §4b). Never scaffold a folder from memory or inference.
- Existing entity `.md` files may be updated when instructed. Increment version.
- `entity.yaml` / `packet.yaml` files may be updated when instructed. These are the machine-readable source of truth — surface proposed changes to Alpha before writing.
- `docs/*.md` files may be written or updated when instructed.
- `decisions/*.md` files: follow the decision recording protocol in §4d.

**`04_docs/`**
- All `.md` files may be written or updated when instructed by a task contract.
- `REPO_PROFILE.md` may only be updated when root folder paths have actually changed. Do not update it speculatively.

**`05_hub/`**
- Agents may create new items in `05_hub/input_hub/raw_drop/` or `05_hub/output_hub/`.
- Stage promotion (raw_drop → triage → candidate → pending → promoted) requires Alpha approval at each gate. Agents do not self-promote items.
- `promoted_log/` — append only. Do not modify existing entries.

**`06_state/`**
- `inputs/*.yaml` — propose changes to Alpha; write only after approval.
- `registers/`, `graph/` — never write directly. Script-managed only.
- `audit/` — append log entries only. Never edit existing entries.

**`07_scripts/`**
- Script files may be modified when instructed by a task contract.
- Scripts are idempotent by design. Do not write scripts that have side effects on second run.

**`08_command/`**
- No writes without an authorising ADR. This zone is deferred.

**`09_runs/`**
- Agents may write freely. This zone is ephemeral and gitignored.

**Root control surfaces (`README.md`, `START_HERE.md`, `AGENTS.md`, `CLAUDE.md`)**
- May only be modified when instructed by a task contract.
- These files govern agent and human operation — changes have immediate effect on all future sessions. Surface the completed file to Alpha before declaring done.

## Prohibited actions

1. Do not rename root folders or change the `NN_` prefix sequence without an authorising ADR approved by Alpha.
2. Do not create files or folders outside the declared structure without explicit Alpha approval. "Outside the declared structure" means any path not resolvable from the path registry in §3.
3. Do not write to `06_state/registers/`, `06_state/graph/`, or `06_state/audit/` (except audit appends) directly. These paths are script-managed.
4. Do not infer `scope_level` from folder path. Read it from the `scope_level` field in front matter.
5. Do not construct file paths from memory or from folder name inference. All paths are derived from the path registry in §3 or from the task contract.
6. Do not modify an ADR with status `active` or `frozen` without Alpha instruction and a version increment.
7. Do not promote hub items through intake stages without Alpha approval at each gate.
8. Do not begin executing a task before completing the cold-start protocol in §2 in full.

## Escalation triggers

Stop and surface to Alpha when any of the following conditions is true. State the trigger name, describe the specific conflict or gap, and ask the minimum question needed to resolve it. Do not propose a resolution — present the conflict and wait for instruction.

1. **Path not in registry** — a required file path cannot be resolved from the path registry in §3 or from the task contract.
2. **Register–entity conflict** — a JSON register in `06_state/registers/` contains data that conflicts with the corresponding `entity.yaml` file.
3. **Missing parent entity** — an entity creation task requires a parent (program needs a portfolio; project needs a program; task packet needs a project) and that parent does not exist.
4. **ADR ID conflict** — the next sequential ADR ID from `01_governance/decision-register.md` conflicts with an existing file ID in any `decisions/` folder.
5. **Task contract ambiguity** — the task contract does not specify a required field, path, or decision, and the correct answer cannot be determined from the pre-flight files with certainty.
6. **Prohibited action required** — completing the task as specified would require an action listed in §7.
7. **Pre-flight file conflict** — two pre-flight files contain contradictory information about the same rule, path, or field.
8. **No task contract** — the session was initiated without a task contract and Alpha has not provided explicit instructions. Do not infer a task.
